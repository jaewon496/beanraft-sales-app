전체 흐름

사용자가 주소를 입력하면 어떤 일이 일어나는가

검색 -> 수집 -> 분석 -> 결과

약 30~60초 동안 55개 API에서 데이터를 수집하고 AI가 종합 분석합니다

STEP 1. 주소 해석 (0~2초)

입력된 도로명 주소를 위도/경도 좌표로 변환합니다. 카카오 지오코딩 API가 "서울시 용산구 청파로 205-6"을 (37.5449, 126.9714) 같은 숫자로 바꿔줍니다.

도로명 주소 입력 -> 카카오 지오코딩 -> 위도/경도 좌표 -> 행정동 코드 확인

STEP 2. 카페 수집 (2~10초)

좌표를 중심으로 반경 500m 안의 모든 카페를 찾습니다. 카카오 로컬 API로 1차 수집하고, 네이버 지역검색으로 실제 존재 여부를 교차 검증합니다.

카카오 로컬 검색 -> 카테고리 필터링 -> 네이버 교차검증 -> 비카페 제거 -> 최종 카페 리스트

미용실, 네일샵 등이 "카페"로 등록된 경우가 많아서 AI(Gemini)가 매장명과 카테고리를 분석해 진짜 카페만 남깁니다.

STEP 3. 상권 데이터 수집 (10~40초)

55개 API에서 유동인구, 매출, 임대료, 배달, 인프라 등의 데이터를 동시에 수집합니다. 서울 지역은 서울 열린데이터 API가 추가로 작동합니다.

STEP 4. AI 분석 + 카드 생성 (40~60초)

수집된 모든 데이터를 Google Gemini AI에게 전달합니다. AI가 14개 카드 각각에 맞는 맞춤형 인사이트를 작성합니다. 단순히 숫자를 보여주는 것이 아니라 "이 지역에서 카페를 열면 어떤 전략이 필요한지"까지 분석합니다.

개별 카페의 월매출을 추정하는 4-Layer 시스템

4-Layer 매출 추정 엔진

카페마다 가용한 데이터가 다르기 때문에, 가장 정확한 방법을 자동 선택합니다

Layer 1 - 건물 직접 매출 (정확도 95%)

조건: 건물에 매장이 1개뿐일 때

원리: OpenUB가 카드사(신한카드 등)의 실제 결제 데이터를 기반으로 건물별 월매출을 산출합니다. 건물에 매장이 하나뿐이면 건물 매출 = 그 매장 매출이므로 가장 정확합니다.

좌표 -> S2 셀 토큰 생성 -> OpenUB bd/hash -> OpenUB gp (건물 목록) -> OpenUB bd/sales (건물 매출) -> 카페와 건물 좌표 매칭

Layer 2 - 건물 내 분배 (정확도 93%)

조건: 건물에 매장이 여러 개일 때

원리: 건물 총매출을 알지만 매장이 5개면 각각이 얼마인지 모릅니다. 이때 Zipf 법칙이라는 통계적 분배 법칙을 적용합니다.

Zipf 법칙: "1등 매장이 2등의 1.7배, 3등의 2.4배 매출을 올린다"는 자연 법칙. 매장의 순위는 카테고리(카페>음식점>소매), 리뷰 수, 영업패턴 유사도로 결정합니다.

핵심 개선점: 건물 전체 매출이 아닌 카페 카테고리(A12) 매출만 추출해서 카페끼리만 분배. 이렇게 하면 음식점/소매점의 매출이 섞이지 않아 정확도가 크게 올라갑니다.

Layer 3 - 프랜차이즈 브랜드 평균 (정확도 90%)

조건: 스타벅스, 투썸플레이스 등 프랜차이즈일 때

원리: 프랜차이즈 본사의 정보공개서에서 가맹점 연평균 매출을 가져옵니다. 이 값의 70%는 브랜드 자체의 힘이고, 30%만 지역 영향을 받는다고 보정합니다.

예: 스타벅스 연매출 7.3억 / 12개월 = 월 6,083만원. 이 지역의 평균 매출이 전국보다 20% 낮다면 -> 6,083 x (1 + (-0.2 x 0.3)) = 5,718만원

Layer 4 - ML 모델 / 경쟁 조정 (정확도 78~85%)

조건: 개인카페이고 건물 매출 데이터가 없을 때 (가장 많은 케이스)

원리: 머신러닝 모델이 4가지 요소를 분석해 동네 평균 대비 이 카페의 매출 수준을 예측합니다.

ML Feature가중치의미
요일별 영업패턴 균등도25%매일 고르게 영업하는 카페 = 매출 안정적
활성 영업일 수35%주 7일 영업 > 주 5일 = 매출 높음
리뷰 비율25%동네 평균 대비 리뷰가 많으면 = 인기 카페
영업 연수15%오래 살아남은 카페 = 매출 안정적

이 4가지 점수를 합산해 0~1 사이 점수를 만들고, 이 점수에 따라 동네 평균 매출의 0.7배~1.5배 범위에서 추정합니다.

OpenUB 연동 방식 (핵심)

OpenUB는 공식 API가 아닌 웹사이트 역분석으로 동작합니다. OpenUB 웹사이트가 내부적으로 호출하는 API를 분석해서 동일한 방식으로 호출합니다.

위도/경도 -> S2 Geometry 셀 토큰 (Level 14) -> bd/hash (셀 -> 해시키) -> gp (해시키 -> 건물 목록+매출) -> bd/sales (건물 ID -> 카테고리별 매출)

카테고리 코드 A12 = 카페/음료. 이 코드로 필터링하면 건물 내 카페 매출만 추출 가능. 인증 없이 무료로 동작합니다.

카페 필터링 파이프라인 (STEP E)

카카오에서 검색된 매장 중 진짜 카페만 골라내는 5단계 필터:

E-1 카테고리 필터 -> E-1.5 중복 제거 -> E-1.7 네이버 교차검증 -> E-2 AI 판별 (Gemini) -> E-3 최종 확정

E-2에서 3가지 자동 판정이 작동합니다:

  • autoKeep: 네이버에서 카페로 확인 + 지역 일치 -> 무조건 카페
  • autoRemove: 네이버 매칭 0건 + 비카페 카테고리 -> 무조건 제거
  • geminiJudge: 애매한 나머지 -> AI가 최종 판단

이 파이프라인의 결과: 같은 주소로 3번 검색해도 카페 수가 동일합니다 (비결정성 해결 완료).
